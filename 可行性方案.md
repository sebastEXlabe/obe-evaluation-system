## 1. 项目概述

### 1.1 项目背景

随着工程教育认证的全面推行，成果导向教育（Outcome-Based Education, OBE）已成为我国高等教育改革的核心指导理念。与此同时，CDIO（Conceive-Design-Implement-Operate）工程教育模式强调"构思—设计—实施—运行"的全生命周期实践能力培养。将 OBE 与 CDIO 有机融合，构建面向软件工程课程的教学评价管理系统，是当前高等工程教育改革的重要方向。

传统的教学评价系统往往是"成绩录入工具"，缺乏对过程性数据的自动采集、对个体贡献的量化评估，以及对课程目标达成度的闭环分析。本项目旨在设计并实现一套融合 OBE 顶层设计与 CDIO 过程管理的软件工程课程教学评价管理系统。

### 1.2 项目目标

1. 构建 OBE 驱动的课程目标→评价指标→数据采集的顶层设计框架
2. 实现 CDIO 四阶段（构思-设计-实施-运行）的项目过程追踪
3. 集成 MaxKB 智能知识库，自动采集学生学习行为数据
4. 建立"小组成果+个人贡献"的多元融合评价体系
5. 实现课程目标达成度的可视化分析与持续改进闭环

## 2. 需求分析

### 2.1 核心痛点

| 痛点 | 描述 | 解决思路 |
|------|------|----------|
| **搭便车问题** | 小组协作中学生贡献度无法量化，存在不作为现象 | 通过 Git 提交记录、角色分工、贡献度系数实现个体量化 |
| **过程性考核缺失** | 日常问答、协作行为等过程数据难以采集 | MaxKB 智能问答自动记录 + Git 平台数据采集 |
| **OBE 达成度难追踪** | 无法从课程目标反向验证教学效果 | 课程目标→指标点→评价→达成度分析的全链条闭环 |

### 2.2 六大功能模块

**模块一：课程目标与评价指标设计（OBE 顶层设计）**

- OBE 三维度设定课程目标（知识、能力、素养）
- 课程目标分解为毕业要求指标点
- 为每个指标点配置评价方式、权重、数据源
- 支持将评价指标映射到 CDIO 四阶段（构思→设计→实施→运行）
- 设计逻辑：先定义"要达成什么目标"→再确定"如何评价"→最后指定"谁采集数据"

**模块二：小组分工与角色分配（CDIO 组织架构）**

- 教师端创建项目小组（建议 4 人/组）
- 内置四种角色：项目经理、平台开发员、测试员、文档管理员
- 为各角色绑定差异化评价指标
- 支持项目不同阶段动态调整分工

**模块三：MaxKB 智能问答与学习行为采集（AI 赋能层）**

- 3.1 学生端：7×24 小时智能问答辅导（MaxKB + 课程知识库），支持自主测验
- 3.2 教师端：AI 学情预警，识别高频疑难知识点，生成班级知识热力图
- 3.3 数据管理：问答记录自动写入过程性评价数据库，计入自主学习能力维度
- MaxKB 定位：不只是"问答工具"，而是评价系统的数据采集前端

**模块四：项目过程追踪与协作管理（CDIO 实施支撑）**

- 项目里程碑按 CDIO 四阶段设定节点与进度跟踪
- 任务看板：按角色分配任务，记录完成状态
- Git 集成：对接 Gitee/GitHub 自动采集 Commit 记录（反搭便车核心证据）
- 项目日志：学生按时提交每日/每周进展
- 贡献度加分记录

**模块五：多元融合评价（OBE-CDIO 评价体系）**

- 5.1 小组成果评价：对源码、文档、测试报告、演示答辩进行小组评分
- 5.2 个人贡献评价（反搭便车）：个人分工贡献分 = 贡献占比 × 小组成果分
- 5.3 过程数据自动采集：MaxKB 问答、Git 提交、自主测验、项目日志
- 5.4 角色专项评价：各角色有独立的评价维度和侧重

**模块六：达成度分析与可视化展示（OBE 收尾环节）**

- 课程目标达成度计算（加权汇总）
- 能力雷达图（五维度：知识掌握、工程实践、团队协作、沟通表达、自主学习）
- 班级/小组达成度分布图
- 自动改进建议：标注达成度偏低的指标点
- 成绩溯源：从达成度下钻到原始评价数据

### 2.3 数据闭环

学生使用 MaxKB 问答 → 系统自动记录行为数据 → 纳入过程性评价 → 期末达成度分析 → 教师改进教学 → 下一轮课程调整

## 5. 技术可行性分析

### 5.1 技术选型

| 层次 | 技术 | 选型理由 |
|------|------|----------|
| **前端** | Vue 3 + TypeScript + Element Plus | 主流框架，生态成熟 |
| **可视化** | ECharts 5.x | 支持雷达图、热力图、分布图 |
| **后端** | Spring Boot 3.x + JDK 17 | 企业级框架，开发效率高 |
| **ORM** | MyBatis-Plus | 简化 CRUD，支持复杂查询 |
| **数据库** | PostgreSQL 15 | JSON字段、窗口函数，适合分析型查询 |
| **缓存** | Redis 7 | 高频统计缓存、会话管理 |
| **文件存储** | MinIO | 开源 S3 兼容对象存储 |
| **AI 知识库** | MaxKB v2.x (GPLv3, 21.6K Stars) | Python/Django, 成熟开源 |
| **LLM** | DeepSeek / 通义千问 | 国产大模型，性价比高 |
| **Git 集成** | Gitee OpenAPI / GitHub REST API | 采集 Commit 记录 |
| **部署** | Docker Compose | 一键部署，环境一致 |

### 5.2 关键集成方案

**MaxKB 集成**：
- MaxKB 独立部署（Docker），提供 RESTful API
- 系统通过 API 管理知识库、获取问答统计
- 问答记录通过 Webhook/定时任务同步至评价数据库
- 可行性：✅ 高（API 完善，文档齐全，21.6K Stars）

**Git 平台数据采集**：
- Gitee OpenAPI v5 / GitHub REST API
- OAuth2 授权后获取 Commit 历史
- 采集维度：提交次数、代码变更、时间分布
- 可行性：✅ 高（API 成熟稳定）

**OBE 达成度计算**：
- 加权综合评价模型，数学基础成熟
- 达成度_i = Σ(w_j × score_j) / Σ(w_j × full_score_j)
- 可行性：✅ 高（算法简单清晰）

### 5.3 技术风险

| 风险 | 等级 | 应对 |
|------|------|------|
| MaxKB API 版本兼容 | 低 | 锁定 v2.x LTS 版本 |
| Gitee API 限流 | 低 | 本地缓存 + 定时增量同步 |
| LLM 服务不稳定 | 中 | 多模型切换支持 |
| 高并发性能 | 低 | 课程级应用，并发可控 |

## 7. 操作可行性分析

### 7.1 用户角色与操作难度

| 角色 | 核心操作 | 难度 |
|------|----------|------|
| 系统管理员 | 系统配置、用户管理、数据维护 | 低 |
| 教师 | 设定目标/指标、创建小组、查看达成度、发起评价 | 低—中（需培训30分钟）|
| 学生 | 加入小组、MaxKB 问答、提交日志、查看评价 | 低（直觉化操作）|

### 7.2 易用性设计

- 主流管理后台交互模式（左侧导航 + 右侧内容区）
- 操作流程与 OBE-CDIO 方法论步骤对齐
- 关键操作提供引导式配置向导
- MaxKB 问答界面类 ChatGPT 对话框，零学习成本

### 7.3 部署运维

- Docker Compose 一键部署（5 容器：Web、DB、Redis、MinIO、MaxKB）
- 数据备份脚本（pg_dump 定时任务）
- 适用于单人运维
